# 아기의 식사 — 서비스 기획서

> **문서 목적**: "아기의 기록" 앱과 디자인·기술 스택을 공유하는 자매 앱 "아기의 식사"의 개발 기획서.
> 다른 개발 세션·타 앱(아기의 기록)에서 연동 스펙 참조용으로 사용할 수 있도록 자기완결형으로 작성함.
> **최종 수정**: 2026-07-23

## 0. 앱 개요

| 항목 | 내용 |
|---|---|
| 앱 이름 | **아기의 식사** (후보: 아기의 밥상, 아기의 한 끼 — 시리즈 네이밍 일관성 우선 시 "아기의 식사" 확정) |
| 포지셔닝 | 아기의 발육·발달단계 평가 → 이유식 재료/레시피 추천 → 냉장고 재고·큐브 관리 → 급여 기록 → **섭취 적정성 평가**의 순환 루프 |
| 자매 앱 | 아기의 기록 (수유·수면·배변·체온 등 일상 기록, PDF 진료 리포트, AI 챗봇) |
| 관계 | 아기 프로필·수유 기록·성장 기록을 **공유 스키마**로 연동. 아기의 기록의 "아기 공유코드" 체계를 그대로 수용 |

### 핵심 사용자 루프
```
발달 평가 → 이유식 단계 판정
    ↓
냉장고 재고 확인 (재료/냉동 큐브/냉장 이유식)
    ↓
레시피 3-모드 추천 (있는 걸로 / 한두 개 더 사서 / 새로 장보고)
    ↓
조리 [만들었어요] → 재료 자동 차감, 완성 이유식 재고 등록
    ↓
급여 기록 (사진 첨부, 섭취량, 반응)
    ↓
수유량(아기의 기록 연동) + 이유식 섭취량 통합
    ↓
키·몸무게·개월수 대비 섭취 적정성 평가 → 다음 추천에 반영
```

## 1. 개발환경 및 기술 스택

아기의 기록과 동일 스택을 유지하여 디자인 시스템·컴포넌트·유틸을 최대한 재사용한다.

| 영역 | 기술 | 비고 |
|---|---|---|
| 마크업 | HTML5 (시맨틱 태그, PWA manifest) | |
| 스타일 | CSS3 — **CSS 변수 디자인 토큰** | 아기의 기록의 `base.css`(색상·라운드·그림자·타이포), `components.css`(카드·버튼·배지·모달·토스트), `category.css`(카테고리 색상 테마), `animations.css` 를 그대로 복사해 base로 사용 |
| 로직 | Vanilla JavaScript (ES6+), 해시 기반 라우터 | `router.js`, `storage.js`, `Toast.js`, `Modal.js`, `ageHelper.js`, `dateHelper.js` 재사용 |
| 레이아웃 | 모바일 우선, max-width **430px** | iPhone/Android 최적화 |
| PWA | manifest.json + Service Worker | 홈 화면 설치, **오프라인에서 냉장고 재고 조회 필수 지원** |
| 저장 | localStorage(설정) + **IndexedDB(주 저장소)** | 재고·기록·사진은 IndexedDB |
| AI | Anthropic Claude API — **Cloudflare Workers 프록시** 경유 | API 키는 프론트에 절대 미포함. 월령+발달단계+재고+알레르기를 System Prompt 컨텍스트로 주입 |
| 음성 입력 | Web Speech API + LLM tool calling | "당근 큐브 6개 소분했어" → 재고 등록. 아기의 기록의 음성 기록 패턴 이식 |
| 카메라 | `input[capture="environment"]` | 끼니 사진 촬영/갤러리 첨부 |
| 알림 | Web Push API + SW (iOS 16.4+, Android Chrome) | 소진기한 임박, 새 재료 3일 관찰 알림 |
| 배포 | Vercel 무료 플랜 | §4 연동 방안 A 채택 시 모노레포 하위 경로 배포 |

### 추천 로직 아키텍처 — 규칙 기반 + LLM 하이브리드 (2층 구조)

LLM 단독 추천은 월령 부적합 재료(꿀, 생우유 등)·재고에 없는 재료가 섞이는 환각 위험이 있으므로:

1. **규칙 필터층 (클라이언트, 결정론적)**
   - 재료 사전 `allowed_by_stage` 테이블: 단계별 허용/금지 재료 하드 필터
   - 알레르기 이력 재료 제외
   - 재고 매칭률 계산, 소진기한 임박 재료 우선순위 가중
2. **LLM 생성층 (Claude)**
   - 필터 통과 후보 안에서만 조합·조리법·추천 근거 생성
   - 시스템 프롬프트에 후보 재료 목록을 명시적으로 제한 (lexicon-constrained 방식)

## 2. 멀티 아기 지원 및 공유코드 연동 ★

### 2.1 멀티 아기 원칙

- 한 계정(기기)에 **여러 아기 프로필** 등록 가능 (쌍둥이, 형제자매).
- **모든 데이터 store는 `babyId`를 키로 분리 저장**한다. (재고 제외 — §2.4 참고)
- 활성 아기 개념: `localStorage.activeBabyId` 에 현재 선택된 아기 저장.
- **아기 전환 UI**: 전 탭 공통 헤더 좌측에 아기 아바타 칩 표시. 탭하면 바텀시트로 아기 목록 + [아기 추가] + [공유코드로 불러오기]. 전환 시 홈·기록·평가 화면이 해당 아기 데이터로 즉시 갱신 (토스트: "○○(으)로 전환했어요").

### 2.2 아기 공유코드 (아기의 기록 체계 수용)

아기의 기록에서 생성한 **아기 공유코드**를 아기의 식사에서 입력하면 해당 아기의 프로필·수유·성장 기록이 연결된다. 반대로 아기의 식사에서 만든 아기도 동일 체계의 공유코드를 발급할 수 있어, 양방향·양육자 간(엄마↔아빠 기기) 공유가 가능하다.

**공유코드 스펙 (아기의 기록과 반드시 동일 포맷 유지):**

| 항목 | 정의 |
|---|---|
| 형식 | 8자리 영숫자 (혼동 문자 I/l/1/O/0 제외), 예: `K7MX-P4RB` (하이픈은 표시용) |
| 매핑 | `shareCode → babyId(UUID)` |
| 발급 | 아기 프로필 화면 [공유코드 발급] — 이미 발급된 코드가 있으면 재사용 표시 |
| 입력 | 온보딩 또는 아기 추가 바텀시트의 [공유코드로 불러오기] |

**연동 시나리오별 동작:**

| 시나리오 | 동작 |
|---|---|
| **같은 기기, 같은 origin** (§4 방안 A) | 공유코드 입력 불필요할 수도 있으나, 명시적 연결 UX를 위해 코드 입력 시 공유 DB의 `babies` store에서 해당 `babyId`를 조회해 즉시 연결. 데이터 복사 없음(참조) |
| **다른 기기 / 다른 origin** | Cloudflare Workers **KV 릴레이**를 통한 1회성 동기화: 발급 측이 코드 발급 시 프로필+최근 수유·성장 기록 스냅샷을 KV에 업로드(TTL 72시간, 코드로만 조회 가능) → 수신 측이 코드 입력 → 다운로드 후 로컬 IndexedDB에 저장. 이후 갱신은 코드 재발급 또는 JSON 내보내기로 수동 동기화 |
| **아기의 기록에 코드 발급 기능이 아직 없는 경우** | 아기의 식사가 위 스펙으로 선제 구현하고, 아기의 기록에 동일 스펙 발급/입력 모듈을 추가 (본 문서 §2.2가 상호 스펙 합의 기준) |

**보안/프라이버시 유의:**
- KV 릴레이의 스냅샷은 코드 없이는 조회 불가, TTL 만료 시 자동 삭제.
- 스냅샷에는 사진 Blob은 포함하지 않는다 (용량·민감도). 프로필 + 수치 기록만.
- 코드 유출 대비: 발급 화면에 [코드 무효화 후 재발급] 버튼 제공.

### 2.3 온보딩 플로우

```
첫 실행
 ├─ [공유코드가 있어요] → 코드 입력 → 아기 프로필 연결/가져오기 → 홈
 └─ [새로 시작해요] → 아기 이름·생년월일·출생 체중/신장 입력 → 홈
(이후 아기 추가는 헤더 아바타 칩 바텀시트에서 동일 분기)
```

### 2.4 멀티 아기와 데이터 스코프

| 데이터 | 스코프 | 이유 |
|---|---|---|
| 프로필, 성장, 발달 체크, 수유·식사 기록, 사진, 평가 결과 | **아기별** (`babyId` 키) | 개인 데이터 |
| 냉장고 재고 (재료·큐브·냉장 이유식) | **가구 공용 (계정 단위)** | 냉장고는 물리적으로 하나. 단, 큐브·냉장 이유식에는 선택적 `forBabyId` 태그를 두어 "○○ 전용" 표시 가능 |
| 레시피 저장함 | 가구 공용 + 아기별 적합 배지 | 같은 레시피도 아기별 단계·알레르기에 따라 적합/부적합 배지가 달라짐 |
| 추천·평가 실행 | 항상 **활성 아기 기준** | 형제가 단계가 다르면 추천 결과가 달라야 함 |

## 3. 데이터 모델 (IndexedDB object stores)

DB 이름: `baby-shared` (공유 store) + `baby-meal` (아기의 식사 전용 store)

### 3.1 공유 store (`baby-shared`) — 아기의 기록과 스키마 합의 필요 ★

```
babies            { babyId(UUID, PK), name, birthDate, birthWeight, birthHeight,
                    shareCode, allergies: [{ingredient, confirmedDate, severity}],
                    currentStage, createdBy: 'record'|'meal', updatedAt }

feedingLogs       { logId(PK), babyId(idx), timestamp,
                    type: 'breast_direct'|'breast_pumped'|'formula'|'solid',
                    amountMl,          // 분유·유축: 실측 / 직수: null
                    durationMin,       // 직수 수유 시간 → 추정용
                    isEstimated: bool, // 직수 추정치 플래그
                    formulaProduct?, formulaConcentration?,
                    solidMealRef?,     // 이유식일 경우 mealLogs 참조
                    source: 'record'|'meal', updatedAt }

growthRecords     { recordId(PK), babyId(idx), measuredDate,
                    weightKg, heightCm, headCm?, source, updatedAt }
```

> **합의 원칙**: 두 앱 모두 이 스키마로 읽고 쓴다. 필드 추가는 하위 호환(옵셔널)으로만.
> `source` 필드로 어느 앱에서 생성됐는지 추적 → 충돌 시 `updatedAt` 최신 우선(last-write-wins).

### 3.2 아기의 식사 전용 store (`baby-meal`)

```
devChecks         { checkId, babyId(idx), checkDate, items: {목가누기, 지지앉기,
                    혀밀어내기반사소실, 음식관심, 손집기…}, judgedStage }

ingredients       { ingredientId, category, name, purchaseDate, openDate?,
                    storageType: 'room'|'fridge'|'freezer',
                    quantity, unit, expiryDate(자동제안+수정가능), forBabyId? }

cubes             { cubeId, ingredientRef, portionedDate, cubeCount,
                    cubeVolumeMl, expiryDate, forBabyId? }

preparedMeals     { mealId, name, recipeRef?, cookedDate, remainingAmount,
                    expiryDate, forBabyId? }

mealLogs          { logId, babyId(idx), timestamp, menuName, recipeRef?,
                    intakeRatio: 'all'|'half'|'little'|'refused',
                    estimatedIntakeG?, estimatedKcal?,      // 레시피 연동 시 자동 계산
                    reaction, adverseFlag: bool, adverseNote?,
                    isNewIngredient: bool, newIngredientName?,
                    photoIds: [{photoId, photoStage: 'before'|'after'}],
                    cubesUsed: [{cubeId, count}] }          // 자동 차감 연동

photos            { photoId, blob(리사이즈본), thumbBlob(320px), takenAt, babyId }

recipes           { recipeId, title, stageTags, ingredients: [{name, amountG}],
                    steps, nutritionEst: {kcal, proteinG, ironMg…},
                    sourceMode: 'stock_only'|'buy_few'|'fresh_shop', savedAt }

assessments       { assessId, babyId(idx), weekOf, intakeKcalAvg, requiredKcal,
                    adequacyPct, milkSolidRatio, llmSummary, createdAt }
```

## 4. 아기의 기록 연동 방식

| | **방안 A — 동일 origin 하위 경로 배포 (권장)** | 방안 B — JSON 내보내기/가져오기 (폴백) |
|---|---|---|
| 구조 | Vercel 한 프로젝트 아래 `/record`, `/meal` 배포 → 같은 origin → IndexedDB `baby-shared` 실시간 공유 | 아기의 기록 [데이터 내보내기] → JSON → 아기의 식사 가져오기 |
| 장점 | 수유 기록 즉시 평가 반영, 완전한 실시간성, `integrationBridge.js` 개념 완성 | 구현 간단, 기존 배포 유지 |
| 단점 | 기존 별도 도메인 배포 시 마이그레이션 공지 + 1회 데이터 이전 필요 | 수동 동기화, 실시간성 없음 |
| 채택 | **기본 방향** | 과도기용 + 타 기기 공유코드 KV 릴레이와 병행 |

## 5. 하단 탭 구성 — 5개

```
┌──────┬──────┬──────────┬──────────┬──────┐
│  홈  │ 냉장고 │ ★레시피★ │ 식사기록 │ 아기 │
└──────┴──────┴──────────┴──────────┴──────┘
              (중앙 강조 탭)
공통 헤더: [아기 아바타 칩 ▾]  앱 타이틀  [알림 벨]
```

### ① 홈 — 오늘의 대시보드
- 상단 카드: 활성 아기 이름·월령·현재 이유식 단계 배지 (초기/중기/후기/완료기)
- **이번 주 식사 적정성 미니 카드**: 필요량 대비 % 게이지 (여유/적정/살펴보기 3색) → 탭하면 아기 탭 평가 섹션으로
- 오늘의 식사 요약: 오늘 수유+이유식 횟수/총량, 다음 식사 예정
- 임박 알림 카드: 소진기한 D-2 이내 큐브·재료·냉장 이유식 → 탭 시 냉장고 해당 항목
- 새 재료 관찰 카드: "소고기 도입 2일차 — 이상반응 없음" (3일 룰 카운트다운)
- 빠른 액션: [오늘 뭐 만들까?] → 레시피 추천 플로우 직행

### ② 냉장고 — 재고 관리 (가구 공용)
- 세그먼트: 전체 / 원재료 / 냉동 큐브 / 냉장 이유식
- 항목 카드: 이름, 수량("6개 × 30ml"), 구매일/소분일/조리일, **D-day 배지**(여유=초록, 임박=주황, 경과=빨강 — 기존 카테고리 색상 테마 재활용), `forBabyId` 태그 시 아기 아바타 미니 표시
- 추가 플로우: 재료명 자동완성(재료 사전) → 보관형태 → 날짜(오늘 기본) → 소분 시 큐브 수·용량. **음성 입력 지원**
- 소진기한 기본값: 재료 사전에서 자동 제안(사용자 수정 가능). 기본값 테이블은 식약처/대한소아과학회 자료 기준으로 검증 후 앱 내 출처 표기 ⚠️ (안전 수치 — 배포 전 검증 필수)
- 큐브 차감: 식사기록의 "큐브 사용" 체크 시 자동 차감 + 수동 조정

### ③ 레시피 — 3-모드 추천 (중앙 탭, 활성 아기 기준)
진입 시 모드 선택:
1. **있는 걸로 만들기** — 재고 100% 매칭. 소진기한 임박 재료 우선 사용 가중
2. **한두 가지만 더 사서** — 부족 재료 ≤2개. 추가 구매 배지 → 탭하면 장보기 리스트 추가
3. **새로 장보고 만들기** — 재고 무관. 단계·계절·최근 미섭취 재료군 기준 + 전체 장보기 리스트 생성

공통:
- 추천 근거 한 줄 상단 표시: "중기 단계 + 냉동 애호박 큐브 D-2 소진 필요 + 철분 보충 시기"
- 레시피 상세: 재료(재고 보유분 ✓), 조리 단계, 큐브 활용법, 보관/소분 안내, **영양 추정치(kcal·단백질·철분)**
- **[만들었어요]** → 사용 재료·큐브 자동 차감 + 완성 이유식 냉장 재고 등록 여부 선택
- 새 재료 포함 시 "첫 도입 재료" 경고 배지 + 3일 관찰 안내
- 알레르기 이력 재료는 후보에서 하드 제외 (활성 아기 기준 — 형제간 상이할 수 있음)

### ④ 식사기록 — 통합 타임라인 + 사진첩
- 세그먼트: **[타임라인 | 사진첩]**
- 타임라인: 날짜별 카드 — **수유 기록(아기의 기록 연동분)과 이유식 기록을 하나의 타임라인에 통합 표시** (수유는 연한 배경, 이유식은 카드형으로 시각 구분)
- 기록 항목: 시간, 메뉴(레시피 연동 or 직접 입력), 섭취량(전량/절반/조금/거부), 반응 메모, 이상반응 플래그(발진·구토·설사 + 사진), 새 재료 여부
- **끼니 사진 첨부**: 카메라 버튼(`capture="environment"`) 즉시 촬영 or 갤러리 선택. 카드에 썸네일, 탭하면 전체 화면 뷰어
  - 저장: canvas 리사이즈(장변 1280px, JPEG/WebP q0.8, ~100–300KB) Blob + 320px 썸네일 별도 → IndexedDB
  - `photoStage: before/after` 구분 필드 — 추후 vision 기반 잔량 추정→섭취량 자동 보정 연구 확장 대비
- 사진첩: 날짜별 이유식 사진 그리드 (성장 기록·공유용)
- 재료 이력 뷰: 도입한 재료 전체 + 첫 도입일·반응 — 알레르기 관리 핵심 화면
- PDF 내보내기: 최근 기록 + 이상반응 이력 리포트 (소아과 진료 연계)

### ⑤ 아기 — 발육·발달·섭취 적정성 평가
- **아기 프로필 목록** (멀티 아기): 아기별 카드, [공유코드 발급/무효화], [공유코드로 불러오기], [아기 추가]
- 성장 기록: 체중·신장 입력 → **WHO/질병관리청 성장도표 백분위 그래프** (기존 체온 그래프 컴포넌트 확장)
- 발달 체크리스트 (이유식 직결 항목): 목 가누기, 지지 앉기, 혀 밀어내기 반사 소실, 음식 관심, 손 집어먹기(핑거푸드 시점)…
- 단계 판정 카드: 월령+발달 체크 종합 → 현재 단계 + 다음 단계 진입 조건. **"권장" 톤 유지, 소아과 상담 권고 문구 고정 노출**
- **식사 평가 섹션** (§6 상세):
  - 주간 섭취 적정성 게이지 (필요량 대비 %)
  - 그래프 1: 일일 총 섭취 추이 — 유즙/이유식 **누적 막대** (비율 변화 가시화)
  - 그래프 2: 성장 백분위 곡선 + 섭취 적정성 구간 배경 음영 오버레이 (시그니처 뷰)
  - LLM 해석 카드 + 상담 권고 문구(고정)
- 설정: 우상단 기어 아이콘 — 알림, 데이터 내보내기/가져오기, 연동 관리

## 6. 섭취 적정성 평가 모듈 (계산층 + 해석층)

### 6.1 계산층 — 클라이언트, 결정론적 (LLM에 수치 계산 위임 금지)

**일일 에너지 섭취 추정:**

| 유형 | 추정 방식 | 불확실성 |
|---|---|---|
| 분유 | ml × 표준 열량계수 (제품별 조정 가능) | 낮음 |
| 유축 모유 | ml 기반 | 낮음 |
| 직수 모유 | 월령별 회당 평균 추정치 (수유 시간 참고) | **높음 — `isEstimated` 플래그, UI에 "추정" 표기** |
| 이유식 | **레시피 연동 시 재료 조성 기반 열량·영양소 산출** × 섭취 비율(전량/절반/조금) | 중간 — 본 앱의 차별점 |

**필요량 산정:**
- 보건복지부 **KDRIs(한국인 영양소 섭취기준)** 영아 에너지 필요추정량 + 체중당 필요량 내장 테이블 ⚠️ 배포 전 최신판 원문 검증 + 출처 표기
- 성장 백분위(체중·신장)와 반드시 결합 해석
- 유즙:이유식 비율 — 단계별 권장 범위와 실제 비교 (대한소아과학회/질병관리청 기준 검증 ⚠️)

### 6.2 해석층 — Claude, 구조화 컨텍스트 주입

입력: `{섭취 추정치, 필요량 대비 %, 최근 2주 추이, 성장 백분위 변화, 유즙/이유식 비율, 단계, 알레르기}`
출력 형식 고정: **"잘 되고 있는 점 / 살펴볼 점 / 다음 단계 제안"**

**가드레일 (시스템 프롬프트 명시):**
- 진단적 표현 금지 ("성장부진입니다" ✗ → "체중 백분위가 2구간 이상 내려왔어요. 소아과 상담을 권해요" ✓)
- 백분위 급락·지속적 섭취 부족 패턴 감지 시 소아과 상담 권고 **고정 출력**
- 직수 추정치 비중이 높은 주간은 "추정 기반" 한계 문구 자동 첨부

## 7. 안전 가드레일 요약

1. 단계별 금지 재료 하드 필터 (꿀 <12개월, 생우유 등) — LLM 이전 단계에서 차단
2. 알레르기 이력 재료 추천 하드 제외 (아기별)
3. 새 재료 3일 관찰 룰 — 배지·알림·홈 카드
4. 보관기한 기본값 출처 표기 + 사용자 수정 가능
5. 의학적 판단 → 소아과 상담 권고 고정 문구 (아기의 기록과 동일 원칙)
6. 공유코드 KV 스냅샷: TTL 72h, 사진 미포함, 무효화/재발급 가능
